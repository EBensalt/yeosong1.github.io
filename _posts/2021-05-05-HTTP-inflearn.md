# HTTP 기본 지식 정리
- 인프런 김영한 님 강의 [모든 개발자를 위한 HTTP 웹 기본 지식](https://www.inflearn.com/course/http-%EC%9B%B9-%EB%84%A4%ED%8A%B8%EC%9B%8C%ED%81%AC/)를 뼈대로 정리

## Q. 인터넷 통신은 어떻게 이루어지나?
**IP(인터넷 프로토콜)** 이라는 통신 규약을 지키면 어떤 길을 통해서 가도 결국 통신 가능

### IP(인터넷 프로토콜) 역할
  - 지정한 IP 주소에 데이터 전달
  - **패킷packet**이라는 통신 단위로 데이터 전달

#### 패킷
출발지 IP, 목적지 IP, 데이터 내용 묶어서 인터넷에 던지면 타고 타고 잘 도착함. (편지봉투에 주소 쓰고 우편 붙혀 보내면 편지가 잘 도착하듯이)

### 💥 IP(인터넷 프로토콜)의 한계 -> 🛠 전송 계층에 TCP, UDP 프로토콜 사용해서 보완 
- **비연결성**
  - 패킷을 받을 대상이 없거나 서비스 불능 상태여도 패킷 전송
- **비신뢰성**
  - 중간에 패킷이 사라지면?
  - 패킷이 순서대로 안오면? (1500바이트쯤 되면 보통 끊어서 보냄)
- **프로그램 구분의 문제**
  - 같은 IP를 사용하는 서버에서 통신하는 애플리케이션이 둘 이상이면?

### TCP 전송 제어 프로토콜(Transmission Control Protocol
연결지향 - TCP 3 way handshake (가상 연결)
• 데이터 전달 보증
• 순서 보장
• 신뢰할 수 있는 프로토콜
• 현재는 대부분 TCP 사용

### UDP 사용자 데이터그램 프로토콜(User Datagram Protocol)
하얀 도화지에 비유(기능이 거의 없음)
• 연결지향 - TCP 3 way handshake X
• 데이터 전달 보증 X
• 순서 보장 X
• 데이터 전달 및 순서가 보장되지 않지만, 단순하고 빠름

정리
• IP와 거의 같다. **+PORT +체크섬 정도만 추가**
• 애플리케이션에서 추가 작업 필요
• HTTP3 에서는 인기 있을지도?

### PORT 
한 대의 기기(아이피) 내에서 진행중인 프로세스들을 구분할 수 있게 번호가 필요하다. 
- 0 ~ 65535 할당 가능한데
- 0 ~ 1023 사이의 Well-known 포트는 사용하지 않는 것이 좋다.
  - 예) FTP - 20, 21 / TELNET - 23 / HTTP - 80 / HTTPS - 443

### DNS - 도메인 명을 IP 주소로 변환
- 숫자로 된 IP 주소를 기억하기 어려운 문제 & IP 주소가 변경될 경우의 문제를 해결

## URI와 웹 브라우저 요청 흐름
### URI - Uniform Resource Identifier
- URL: Uniform Resource Locator 리소스가 있는 위치를 지정
- URN: Uniform Resource Name 리소스에 이름을 부여. 불편하니까 거의 안씀 ex) urn:isbn:8960777331

URI 포맷 스트링: `scheme://[userinfo@]host[:port][/path][?query][#fragment]`<br>
예시 :         `https://www.google.com:443/search?q=hello&hl=ko`

• scheme 예) https://
  - 자원에 접근하는 방식에 대한 규약
• [userinfo@] 
  - URL에 사용자정보를 포함해서 인증할 때 사용. 거의 안씀.
- host 예) www.google.com
  - 도메인 명. IP 주소 직접 입력도 가능
• [:port] 예) :443
  - 일반적으로 생략.
• [/path] 예) /search
  - 리소스의 경로이다. 계층적 구조.
• [query 파라미터, 쿼리 스트링] 예) q=hello&hl=ko)
  - 키=밸류 형태
  - `&`으로 추가 가능
- [fragment] html 내부 북마크.

### 웹 브라우저 요청 흐름

0. 내가 브라우저에 `https://www.google.com:443/search?q=hello&hl=ko`를 입력한다.
1. DNS 조회
2. 웹 브라우저가 HTTP 요청 메시지를 생성한다.

~~~
  GET /search?q=hello&hl=ko HTTP/1.1
  Host: www.google.com
  ...
~~~
2. 소켓 라이브러리를 통해 전달
3. TCP/IP 계층에서 패킷 한번 더 포장
4. 인터페이스 계층에서 또 포장
5. 서버 도착. 반대로 하나씩 개봉, 요청 분석 후 응답 메시지 생성.
6. 같은 방법으로 클라이언트(브라우저)에 전달
7. 브라우저가 내용을 어떻게 잘 렌더링해서 띄워줌

## HTTP 기본

### 인트로: 모든 것이 HTTP
**HTTP 메시지에 모든 것을 전송하는 요즘**
• HTML, TEXT
• IMAGE, 음성, 영상, 파일
• JSON, XML (API)..
• 거의 모든 형태의 데이터 전송 가능
• 서버간에 데이터를 주고 받을 때도 대부분 HTTP 사용
- TCP에 직접 연결하는 경우는 게임 서버? 특수한 경우 밖에 없음
- 모바일 게임은 HTTP 통신도 많이 씀

**HTTP의 간략한 역사**
• HTTP/0.9 1991년: GET 메서드만 지원, HTTP 헤더X
• HTTP/1.0 1996년: 메서드, 헤더 추가
• HTTP/1.1 1997년: 가장 많이 사용, 우리에게 가장 중요한 버전
  • RFC2068 (1997) -> RFC2616 (1999) -> RFC7230~7235 (2014)
• HTTP/2 2015년: 성능 개선
• HTTP/3 진행중: TCP 대신에 UDP 사용, 성능 개선

**프로토콜별 기반 버전**
- TCP: HTTP/1.1, HTTP/2
• UDP: HTTP/3
• 현재 HTTP/1.1 주로 사용
• HTTP/2, HTTP/3 도 점점 증가
• 보통 웹 페이지에는 이게 다 섞여 있는 듯..

**HTTP 특징 요약**
• 클라이언트-서버 구조
• 무상태 프로토콜(스테이트리스), 비연결성
• HTTP 메시지
• 단순함, 확장 가능

### 클라이언트 서버 구조
클라와 서버를 분리를 한다는 개념이 중요함.<br>
각자 독립적으로 진화 가능한 것이 장점.

### Stateful = 상태 유지, Stateless = 무상태 프로토콜
- Stateful 상태 유지
- Stateless 무상태 프로토콜: 전에 누가 뭘 했는지 문맥을 **기억하지 않음**
  - -> 중간에 다른 서버가 응답해도 응답만 잘하면 문제 없음!
  - -> 갑작스런 클라이언트 증가에도 서버 무한 증설로 대응이 가능
  - -> **스케일 아웃(수평 확장)에 유리**

**최대한 스테이트리스하게 설계하는게 다음과 같은 상황에 좋다: 정말 같은 시간에 딱 맞추어 발생하는 대용량 트래픽**
• 예) 선착순 이벤트, 명절 KTX 예약, 학과 수업 등록
• 예) 저녁 6:00 선착순 1000명 치킨 할인 이벤트 -> 수만명 동시 요청
첫 페이지에 정적 html 한 장 넣고 거기서 볼거리를 넣어서 조금 시간 끈 다음 들어오도록 유도하는 편이라는 듯..

**Stateless 실무 한계?**
• 모든 것을 무상태로 설계 할 수 있는 경우도 있고 없는 경우도 있다.
• 무상태 - 예) 로그인이 필요 없는 단순한 서비스 소개 화면
• 상태 유지 - 예) 로그인
• 로그인한 사용자의 경우 로그인 했다는 상태를 서버에 유지
• 일반적으로 브라우저 쿠키와 서버 세션등을 사용해서 상태 유지
• 상태 유지는 최소한만 사용



### 비 연결성(connectionless)

**장점**
HTTP는 기본이 연결을 유지하지 않는 모델.
• 일반적으로 초 단위의 이하의 빠른 속도로 응답
• 1시간 동안 수천명이 서비스를 사용해도 실제 서버에서 동시에 처리하는 요청은 수십개 이하로 매우 작음
• 예) 웹 브라우저에서 계속 연속해서 검색 버튼을 누르지는 않는다.
• 서버 자원을 매우 효율적으로 사용할 수 있음

**한계와 극복**
• 페이지 넘어갈 때 TCP/IP 연결을 새로 맺어야 함 - 3 way handshake 시간이 계속 추가되는 문제
• 웹 브라우저로 사이트를 요청하면 HTML 뿐만 아니라 자바스크립트, css, 추가 이미지 등등 수 많은 자원이 함께 다운로드
• 지금은 HTTP 지속 연결(Persistent Connections)로 문제 해결
• HTTP/2, HTTP/3에서 더 많은 최적화

**예전에는 keep-alive 사용하다가 지금은 지속 연결(Persistent Connections)을 기본으로 함.**

### HTTP 메시지
요청 메시지와 응답 메시지의 구조

~~~~
                  요청 메시지 예시                              응답 메시지 예시

start-line   ->   GET /search?q=hello&hl=ko HTTP/1.1        HTTP/1.1 200 OK
header       ->   Host: www.google.com                      Content-Type: text/html;charset=UTF-8
                                                            Content-Length: 3423
empty line   ->   \r\n                                      \r\n
message body ->                                             <html>
                                                              <body>...</body>
                                                            </html>
~~~

**참고2** : 임의의 헤더 필드 생성 가능, 그러나 물론 이 약속을 아는 클라이언트만 사용할 수 있겠죠

## HTTP 메서드

개괄

• GET: 리소스 조회
• POST: 요청 데이터 처리, 주로 등록에 사용
• PUT: 리소스를 대체, 해당 리소스가 없으면 생성
• PATCH: 리소스 부분 변경
• DELETE: 리소스 삭제
• HEAD: GET과 동일하지만 메시지 부분을 제외하고, 상태 줄과 헤더만 반환
• OPTIONS: 대상 리소스에 대한 통신 가능 옵션(메서드)을 설명(주로 CORS에서 사용)
• CONNECT: 거의 안씀, 대상 자원으로 식별되는 서버에 대한 터널을 설정
• TRACE: 거의 안씀, 대상 리소스에 대한 경로를 따라 메시지 루프백 테스트를 수행

### HTTP API를 만들어보자

좋은 URI의 핵심은 리소스 식별이다.
  - **참고**: 최근에는 리소스라고 안 부르고 represetation이라고 부르기로 명세가 바뀜.

~~~
URI 작명시 유의 사항

조회한다, 등록한다 이런 동사-행위 말고   --->    명사(회원)-리소스에 집중하기

/member-create                           /members/{}
/member-search                           /members/{}
~~~

### HTTP 메서드 GET

• 리소스 조회
• 서버에 전달하고 싶은 데이터는 query(쿼리 파라미터, 쿼리 스트링)를 통해서 전달
• 최근 스펙에는 메시지 **바디**를 사용해서 데이터를 **전달할 수 있지만**
  - 지원하지 않는 서버가 많으므로 **권장하지 않음**


### HTTP 메서드 POST

요청 데이터 처리
- 주로 리소스 등록
• **메시지 바디를 통해 서버로 요청 데이터 전달**
• 서버는 요청 데이터를 **처리**
• 메시지 바디를 통해 들어온 데이터를 처리하는 모든 기능을 수행한다.
• 주로 전달된 데이터로 신규 리소스 등록, 프로세스 처리에 사용
  - 등록하면 보통 201 created로 보내주면서 로케이션(여기에 생겼다고)도 같이 보내준다. 

**요청 데이터를 어떻게 처리한다는 뜻일까?**
[스펙](https://tools.ietf.org/html/rfc7231#section-4.3.3): POST 메서드는 대상 리소스가 리소스의 고유한 의미 체계에 따라 요청에 포함된 표현을 처리하도록 요청합니다.
<br><br>
예를 들어 POST는 다음과 같은 기능에 사용됩니다.
• HTML 양식에 입력 된 필드와 같은 데이터 블록을 데이터 **처리 프로세스에 제공**
  • 예) HTML FORM에 입력한 정보로 회원 가입, 주문 등에서 사용
• 게시판, 뉴스 그룹, 메일링 리스트, 블로그 또는 유사한 기사 그룹에 메시지 게시
  • 예) 게시판 글쓰기, 댓글 달기
• 서버가 아직 식별하지 않은 새 리소스 생성
  • 예) 신규 주문 생성
• 기존 자원에 데이터 추가 (**append**)
  • 예) 한 문서 끝에 내용 추가하기

정리: 이 리소스 URI에 POST 요청이 오면 요청 데이터를 어떻게 처리할지 리소스마다 따로 정해야 함 -> 정해진 것이 없음

1. 새 리소스 생성(등록)
  • 서버가 아직 식별하지 않은 새 리소스 생성
2. 요청 데이터 처리
  • 단순히 데이터를 생성하거나, 변경하는 것을 넘어서 프로세스를 처리해야 하는 경우
    • 예) 주문에서 결제완료 -> 배달시작 -> 배달완료 처럼 단순히 값 변경을 넘어 프로세스의 상태가 변경되는 경우
  • POST의 결과로 새로운 리소스가 생성되지 않을 수도 있음
    • 예) POST /orders/{orderId}/start-delivery (이렇게 어쩔 수 없이 동사를 사용한 URI를 '**컨트롤 URI**'라고 부름)
3. 다른 메서드로 처리하기 애매한 경우
    • 예) JSON으로 조회 데이터를 넘겨야 하는데(메세지를 바디에 넣고 싶은데), GET 메서드를 사용하기 어려운 경우
  • 애매하면 POST

POST는 모든 걸 할 수 있음.
<br>근데 보통 서버에는 **GET으로 오면 캐싱을 하겠다**는 약속이 있어서
<br>조회는 GET으로 하는게 유리함.

### HTTP 메서드 PUT

**리소스를 대체**
• 리소스가 있으면 삭제 후 완전 대체 (리소스 부분 변경은 PATCH로!)
• 리소스가 없으면 생성

  - = POST와 차이점!! POST는 클라이언트가 리소스가 어디에 생길지 모르고 요청함

• A: 안전은 해당 리소스만 고려한다. 그런 부분까지 고려하지 않는다.

#### 멱등(Idempotent Methods)하다 = 한 번 호출하든 두 번 호출하든 100번 호출하든 결과가 똑같다

• f(f(x)) = f(x)
• GET, HEAD, PUT, DELETE..
• GET: 한 번 조회하든, 두 번 조회하든 같은 결과가 조회된다.
• PUT: 결과를 대체한다. 따라서 같은 요청을 여러번 해도 최종 결과는 같다.
• DELETE: 결과를 삭제한다. 같은 요청을 여러번 해도 삭제된 결과는 똑같다.
• *POST: 멱등이 아니다! 두 번 호출하면 같은 결제가 중복해서 발생할 수 있다.*

**멱등 같은 특징은 어디에 쓰는가?**
• 자동 복구 메커니즘 만들 때!!
• 서버가 TIMEOUT 등으로 정상 응답을 못주었을 때, 클라이언트가 같은 요청을 다시 해도 되는가?에 판단 근거!

**Q: 재요청 중간에 다른 곳에서 리소스를 변경해버리면?**
• 사용자1: GET -> username:A, age:20
• 사용자2: PUT -> username:A, age:30
• 사용자1: GET -> username:A, age:30 -> 사용자2의 영향으로 바뀐 데이터 조회
**A: 멱등이라는 요소를 생각할 때 외부 요인으로 중간에 리소스가 변경되는 것 까지는 고려하지는 않는다.**
<br>(물론 서버가 저런 부분을 캐치해서 처리를 해야할 것이다.)

#### 캐시가능(Cacheable Methods)? = 응답 결과 리소스를 캐시해서 사용해도 되는가?

• 스펙상으론 GET, HEAD, POST, PATCH 캐시가능
• **실제로는 GET, HEAD 정도만 캐시로 사용**
• POST, PATCH는 본문 내용까지 캐시 키로 고려해야 하는데, 구현이 쉽지 않음

## HTTP 메서드 활용

### 클라이언트에서 서버로 데이터 전송

데이터 전달 방식은 크게 2가지

• **쿼리 파라미터를 통한 데이터 전송**
  • GET
  • 주로 정렬 필터(검색어)
• **메시지 바디를 통한 데이터 전송**
  • POST, PUT, PATCH 주로 사용
  • 회원 가입, 상품 주문, 리소스 등록, 리소스 변경

4가지 상황 예시
1. **정적 데이터 조회 GET**
  • 이미지, 정적 텍스트 문서
  - 단순하게 리소스 URI 경로를 전달.. 보통 쿼리 파라미터도 필요 없음

2. **동적 데이터 조회 GET**
  • 주로 검색, 게시판 목록에서 정렬 필터(검색어)
  • 조회 조건을 줄여주는 필터, 조회 결과를 정렬하는 정렬 조건에 주로 사용
  • URI + 쿼리 파라미터 사용해서 데이터를 전달

3. **HTML Form 태그를 통한 데이터 전송**
  • 회원 가입, 상품 주문, 데이터 변경
HTML Form submit시 POST 전송
    • 예) 회원 가입, 상품 주문, 데이터 변경
• Content-Type: application/x-www-form-urlencoded 사용
  • form의 내용을 메시지 바디를 통해서 전송(key=value, 쿼리 파라미터 형식)
  • 전송 데이터를 url encoding 처리
    • 예) abc김 -> abc%EA%B9%80
• HTML Form은 GET 전송도 가능
• Content-Type: multipart/form-data
  • 파일 업로드 같은 바이너리 데이터 전송시 사용
  • 다른 종류의 여러 파일과 폼의 내용 함께 전송 가능(그래서 이름이 multipart)
• 참고: HTML Form 전송은 GET, POST만 지원


4. **HTTP API를 통한 데이터 전송**
  • 회원 가입, 상품 주문, 데이터 변경
  • 서버 to 서버, 앱 클라이언트, 웹 클라이언트(Ajax)
